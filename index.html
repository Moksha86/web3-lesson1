Lesson Introduction & Objectives

Lesson Introduction & Objectives

Cryptography is the substrate of trust in Web3. Wallets derive identity from public keys, signatures express consent and authorization, and ciphers preserve confidentiality across peer‑to‑peer links and decentralized storage. This lesson develops a rigorous mental model for the two primary families—symmetric and asymmetric cryptography—then shows how real systems combine them in hybrid patterns. You will see how AES‑GCM and ChaCha20‑Poly1305 deliver fast, authenticated encryption, why RSA and ECC (especially secp256k1) enable open verification, and how wallet‑ and contract‑level capabilities map onto these primitives. We emphasize practical security—nonce management, padding choices, deterministic signing, threat modeling—so you can implement robust flows in production dApps and services.

By the end, you will be able to evaluate trade‑offs (speed, bandwidth, UX, auditability), select suitable parameters, implement reference‑grade Node.js code, and articulate why hybrid encryption is the default architecture: asymmetric steps to establish identity and exchange keys; symmetric steps to protect data efficiently. Each slide pairs theory with a concrete Web3 scenario, a concise code example, and a table or mini diagram to reinforce the model. Quizzes help you self‑assess and prepare for advanced topics like threshold signatures, account abstraction, and post‑quantum migration paths.

Slide 1

Slide 1 — Symmetric vs Asymmetric: The Big Picture

Symmetric and asymmetric cryptography are complementary. Symmetric ciphers use one secret key for both encryption and decryption, yielding exceptional throughput and energy efficiency—ideal for streaming, backups, and large payloads. Their main operational challenge is key distribution: all authorized parties must possess the same secret, and sharing and storing that secret safely is a central risk. Asymmetric cryptography assigns a private key to the owner and a mathematically linked public key to the world. Anyone can encrypt to you (using your public key) or verify your signatures; only you can decrypt or produce valid signatures. This solves distribution and enables decentralized verification, but at a higher computational cost per operation.

PropertySymmetricAsymmetric
KeysSingle shared secretPublic/Private pair
SpeedVery fast; constant per‑byte costSlower; higher per‑operation cost
DistributionChallenging (secure sharing)Straightforward (share public key)
Best UseBulk data encryptionIdentity, signatures, key exchange
Mini Quiz: Why do most secure protocols use a hybrid approach?
Hybrids deliver public verifiability and safe key establishment (asymmetric) with high‑throughput data protection (symmetric), achieving both usability and performance.
Slide 2

Slide 2 — Symmetric Encryption Theory (AES, ChaCha20, 3DES)

Symmetric cryptography encrypts and decrypts with the same key. AES operates on 128‑bit blocks with key sizes of 128/192/256 bits; security in practice hinges on mode of operation. AEAD modes such as AES‑GCM and ChaCha20‑Poly1305 provide confidentiality and integrity in one step. GCM combines counter‑mode encryption with GHASH authentication; ChaCha20‑Poly1305 pairs a stream cipher with a MAC and often excels on devices lacking AES‑NI hardware acceleration. 3DES is legacy due to small block size and poor performance—avoid it in new designs.

CipherMode/SuiteStrengthsCaveats
AES‑256‑GCMAEAD (counter+GHASH)Fast with AES‑NI; integrated integrityStrict nonce uniqueness
ChaCha20‑Poly1305AEAD (stream+MAC)Great on mobile/ARMNonce discipline still required
3DESLegacy block cipherCompatibilitySlow; avoid in new designs
Mini Quiz: Why is nonce reuse fatal for GCM?
It reveals relations between keystream blocks and undermines authentication—attackers can infer plaintext differences and potentially forge tags.
Slide 3

Slide 3 — Symmetric Encryption Code Example (Node.js, AES-256-GCM)

Node.js exposes AES‑GCM via the crypto module. Use a 32‑byte key (AES‑256), a 12‑byte random IV, and AAD to bind context (domain, user IDs, sequence). Store IV and tag with the ciphertext; verify the tag before using the plaintext. Chunk large files into records with unique IVs; bind sequence numbers in AAD to resist truncation and reordering. 


// Node.js AES-256-GCM reference
import { randomBytes, createCipheriv, createDecipheriv } from "crypto";

export function aesGcmEncrypt(plaintext: Buffer, key: Buffer, aad?: Buffer) {
  const iv = randomBytes(12); // 96-bit IV recommended
  const cipher = createCipheriv("aes-256-gcm", key, iv, { authTagLength: 16 });
  if (aad) cipher.setAAD(aad, { plaintextLength: plaintext.length });
  const ciphertext = Buffer.concat([cipher.update(plaintext), cipher.final()]);
  const tag = cipher.getAuthTag();
  return { iv, ciphertext, tag };
}

export function aesGcmDecrypt(
  ciphertext: Buffer, key: Buffer, iv: Buffer, tag: Buffer, aad?: Buffer
) {
  const decipher = createDecipheriv("aes-256-gcm", key, iv, { authTagLength: 16 });
  if (aad) decipher.setAAD(aad, { plaintextLength: ciphertext.length });
  decipher.setAuthTag(tag);
  const plaintext = Buffer.concat([decipher.update(ciphertext), decipher.final()]);
  return plaintext;
}

// Usage
const key = randomBytes(32); // AES-256
const aad = Buffer.from("context:web3-odyssey:lesson-1.1");
const message = Buffer.from("Confidential payload");
const { iv, ciphertext, tag } = aesGcmEncrypt(message, key, aad);
const recovered = aesGcmDecrypt(ciphertext, key, iv, tag, aad);
console.log(recovered.toString()); // "Confidential payload"

Random 12‑byte IVRequired uniqueness for GCMPrevents keystream reuse

Mini Quiz: What belongs in AAD for a dApp message?
Domain/version, sender and recipient identifiers (addresses), content‑type, and sequence numbers—anything you must validate but need not encrypt.
Slide 4

Slide 4 — Symmetric Pros & Cons

Symmetric cryptography wins on speed, latency, and energy. With hardware acceleration (AES‑NI), AES‑GCM can reach multi‑GB/s per core; ChaCha20‑Poly1305 excels on mobile and embedded CPUs. AEAD simplifies integrity by design. The main risk is operational: compromise of the shared key collapses confidentiality and integrity. In Web3, symmetric encryption is a natural fit for encrypting IPFS content, bundle archives, and P2P streaming once access is established via an asymmetric layer.

AspectProsCons
PerformanceExtremely fastCan mask misuse without AEAD
SecurityAEAD gives integrity+authenticityShared key is a liability
OperationsSimple APIsNonce/key lifecycle discipline required
Web3 FitGreat for off‑chain confidentialityNeeds asymmetric layer for access control
Mini Quiz: What is the largest risk in symmetric deployments?
Key distribution and storage. If the shared key leaks, attackers can decrypt and forge valid ciphertexts.
Slide 5

Slide 5 — Asymmetric Encryption Theory (RSA, ECC)

Asymmetric cryptography pairs a private key with a public key. RSA’s security stems from factoring hardness; robust use wraps encryption with OAEP and signatures with PSS. ECC relies on the elliptic curve discrete logarithm problem; ECDSA (secp256k1) provides compact signatures and efficient verification with small keys—ideal for blockchain. In Web3, asymmetric cryptography is identity: private keys authorize actions; public keys enable permissionless verification on and off chain.

CriterionRSAECC (secp256k1)
Key/Signature SizeLargerSmaller (bandwidth‑friendly)
PerformanceMature toolingEfficient verify; wallet‑standard
Use in Web3Key wrapping, legacyWallets, signatures, on‑chain verify
Mini Quiz: Why is ECC often preferred over RSA today?
Comparable security with smaller keys and signatures, improving performance and bandwidth—especially important for blockchain and mobile clients.
Slide 6

Slide 6 — Asymmetric Code Examples (Node.js: RSA-OAEP, ECDSA)

Node.js provides primitives for RSA and EC keys via crypto. Use RSA‑OAEP (SHA‑256) to encrypt small secrets (like AES keys). Use ECDSA on secp256k1 for signatures aligned with Web3 wallets. Ethereum clients sign messages and typed data; contracts verify by recovering the signer. 


// RSA-OAEP (SHA-256) — Encrypt with public, decrypt with private
import { generateKeyPairSync, publicEncrypt, privateDecrypt, constants } from "crypto";

const { publicKey, privateKey } = generateKeyPairSync("rsa", {
  modulusLength: 2048,
  publicKeyEncoding: { type: "spki", format: "pem" },
  privateKeyEncoding: { type: "pkcs8", format: "pem" },
});

const plaintext = Buffer.from("Only the private key holder reads this.");
const ciphertext = publicEncrypt(
  { key: publicKey, padding: constants.RSA_PKCS1_OAEP_PADDING, oaepHash: "sha256" },
  plaintext
);
const recovered = privateDecrypt(
  { key: privateKey, padding: constants.RSA_PKCS1_OAEP_PADDING, oaepHash: "sha256" },
  ciphertext
);
console.log(recovered.toString()); // original message
          

// ECDSA (secp256k1) — Sign with private, verify with public
import { generateKeyPairSync, createSign, createVerify } from "crypto";

const { publicKey: ecPub, privateKey: ecPriv } = generateKeyPairSync("ec", {
  namedCurve: "secp256k1",
  publicKeyEncoding: { type: "spki", format: "pem" },
  privateKeyEncoding: { type: "pkcs8", format: "pem" },
});

const data = Buffer.from("Message bound to my Web3 identity");
const signer = createSign("SHA256");
signer.update(data); signer.end();
const signature = signer.sign(ecPriv);

const verifier = createVerify("SHA256");
verifier.update(data); verifier.end();
const valid = verifier.verify(ecPub, signature);
console.log({ valid }); // { valid: true }
PracticeReasonOutcome
Use OAEP/PSS for RSARandomization; CCA resistanceSafer than textbook RSA
Domain‑separate signaturesPrevent replay across contextsUX clarity; phishing reduction
Prefer ECC for walletsSmall keys; fast verifyCheaper bandwidth/on‑chain costs
Mini Quiz: Why use RSA‑OAEP to wrap a symmetric key?
OAEP randomizes encryption, preventing deterministic outputs and chosen‑ciphertext attacks. It’s ideal for encrypting small secrets like AES keys to a recipient’s public key.
Slide 7

Slide 7 — Asymmetric Pros & Cons

The superpower of asymmetric cryptography is global verifiability: anyone with a public key can validate a signature without contacting the signer or sharing secrets—perfect for decentralized systems. Costs include heavier computation and more nuanced implementation to avoid side channels or misuse. RSA remains ubiquitous in enterprise; ECC powers wallets and modern protocols where artifact size and verification speed matter most.

AspectProsCons
IdentityPublic verifiability; no pre‑shared secretsRotation/revocation policy needed
PerformanceGood for auth/exchangeToo slow for bulk encryption
ArtifactsECC: compact signatures/keysRSA: larger sizes, more bandwidth
Web3 FitWallets, signatures, key wrappingAvoid for large payloads
Mini Quiz: What property makes signatures ideal for decentralized verification?
Anyone can verify with only the signer’s public key—no shared secrets—enabling trust‑minimized, global validation.
Slide 8

Slide 8 — Web3 Use Cases: Wallets, Smart Contracts, Signatures

Wallets manage asymmetric keys: the private key never leaves the user’s secure environment, while the public key (or its hash) becomes the address. Off‑chain, applications request signatures for typed data (EIP‑712) to bind user intent; on‑chain, contracts verify that signature or recover the signer. For confidentiality, apps encrypt user data off‑chain with symmetric ciphers and gate access by encrypting the symmetric key for recipients’ public keys or using threshold schemes. This preserves consent, privacy, and scalability.

PatternSymmetric RoleAsymmetric Role
Off‑chain storageEncrypt blobs with AES‑GCMWrap AES key for recipients
TransactionsN/ASign with ECDSA; verify on‑chain
P2P chatStream frames with AEADPerform key exchange; rotate
Mini Quiz: Why prefer EIP‑712 typed data for signatures?
Structured fields and a domain separator prevent ambiguity and replay across domains, improving both security and UX.

Slide 9 — Hands-On Exercise + Quiz

Implement a hybrid pipeline: (1) Generate a random 32‑byte AES key and encrypt a message with AES‑GCM including AAD. (2) Generate an RSA key pair and wrap the AES key with RSA‑OAEP (SHA‑256). (3) Generate an EC key pair (secp256k1) and sign SHA‑256(ciphertext) for non‑repudiation. (4) Verify the signature, unwrap the AES key with RSA, then decrypt and compare to the original plaintext.

StepActionSuccess Criteria
Key GenCreate AES, RSA, and EC keysPEM outputs for RSA/EC; 32‑byte AES
EncryptAES‑GCM + AADNon‑empty ciphertext and 16‑byte tag
WrapRSA‑OAEP encrypt AES keyUnwrap equals original AES key
SignECDSA over SHA‑256(ciphertext)Verification returns true
RecoverUnwrap → decryptPlaintext equals original
Mini Quiz: Why sign the ciphertext hash instead of plaintext?
It binds the signature to the encrypted content, enabling integrity checks before key unwrapping and avoiding plaintext exposure during verification.
Slide 10

Slide 10 — Summary & Further Reading

Symmetric ciphers (AES‑GCM, ChaCha20‑Poly1305) deliver fast, authenticated encryption for bulk data. Asymmetric primitives (RSA‑OAEP, ECDSA) enable safe key distribution and public verification. The hybrid architecture—negotiate or wrap with asymmetric, protect with symmetric—scales trust and performance simultaneously. Enforce nonce uniqueness, domain‑separate signatures, rotate keys, validate parameters, and keep audit trails.

Final Quiz: Choose the right tool
• Encrypting a 50 MB artifact for IPFS: AES‑256‑GCM (symmetric).
• Sharing the decryption key with a teammate: RSA‑OAEP or ECIES (asymmetric).
• Proving consent to a dApp action: ECDSA signature (asymmetric).
• Live P2P chat: Symmetric AEAD with rotating nonces and sequence AAD.
