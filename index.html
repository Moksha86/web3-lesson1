Slide 1 — Symmetric vs Asymmetric: The Big Picture

Symmetric ciphers use one secret key for encryption and decryption, delivering exceptional throughput and energy efficiency. Asymmetric cryptography separates a private key (kept secret) from a public key (shared widely), enabling open verification and solving the key distribution problem at higher computational cost. Modern systems combine both: asymmetric steps establish identity and secrets; symmetric steps protect data efficiently at scale.

PropertySymmetricAsymmetric
KeysSingle shared secretPublic/Private pair
SpeedVery fast per byteSlower per operation
DistributionHard (share safely)Easy (share public key)
Best UseBulk data encryptionIdentity, exchange, signatures
Mini Quiz: Why do most secure protocols use a hybrid approach?
It combines public verifiability and safe key establishment (asymmetric) with high‑throughput protection (symmetric).

Slide 2 — Symmetric Encryption Theory (AES, ChaCha20, 3DES)

AES‑GCM and ChaCha20‑Poly1305 are AEAD suites combining encryption with authentication. GCM uses counter‑mode and GHASH; ChaCha20‑Poly1305 pairs a stream cipher with a MAC and often excels on devices without AES‑NI. 3DES is legacy and not recommended for new designs.

CipherModeStrengthsCaveats
AES‑256‑GCMAEADVery fast w/ AES‑NIStrict nonce uniqueness
ChaCha20‑Poly1305AEADGreat on mobile/ARMNonce discipline required
3DESLegacyCompatibilitySlow; 64‑bit blocks
Mini Quiz: Why is nonce reuse fatal for GCM?
It reveals keystream relations and undermines authentication; attackers may infer plaintext or forge tags.

Slide 3 — Symmetric Encryption Code Example (Node.js, AES‑256‑GCM)


// Node.js AES-256-GCM reference
import { randomBytes, createCipheriv, createDecipheriv } from "crypto";

export function aesGcmEncrypt(plaintext: Buffer, key: Buffer, aad?: Buffer) {
  const iv = randomBytes(12); // 96-bit IV recommended
  const cipher = createCipheriv("aes-256-gcm", key, iv, { authTagLength: 16 });
  if (aad) cipher.setAAD(aad, { plaintextLength: plaintext.length });
  const ciphertext = Buffer.concat([cipher.update(plaintext), cipher.final()]);
  const tag = cipher.getAuthTag();
  return { iv, ciphertext, tag };
}

export function aesGcmDecrypt(ciphertext: Buffer, key: Buffer, iv: Buffer, tag: Buffer, aad?: Buffer) {
  const decipher = createDecipheriv("aes-256-gcm", key, iv, { authTagLength: 16 });
  if (aad) decipher.setAAD(aad, { plaintextLength: ciphertext.length });
  decipher.setAuthTag(tag);
  const plaintext = Buffer.concat([decipher.update(ciphertext), decipher.final()]);
  return plaintext;
}
Mini Quiz: What belongs in AAD for a dApp message?
Domain/version, sender/recipient addresses, content type, and sequence numbers—validated but not encrypted.

Slide 5 — Asymmetric Encryption Theory (RSA, ECC)

RSA security relies on factoring hardness and uses OAEP/PSS in practice. ECC relies on the elliptic curve discrete logarithm problem; ECDSA on secp256k1 powers Ethereum wallets with compact signatures and efficient verification. Use asymmetric crypto for identity, signatures, and secure key exchange.

Mini Quiz: Why is ECC often preferred over RSA for wallets?
Comparable security with smaller keys and signatures, improving bandwidth and verification costs.

Slide 6 — Asymmetric Code Examples (Node.js: RSA‑OAEP, ECDSA)


// RSA-OAEP (SHA-256)
import { generateKeyPairSync, publicEncrypt, privateDecrypt, constants } from "crypto";

const { publicKey, privateKey } = generateKeyPairSync("rsa", {
  modulusLength: 2048,
  publicKeyEncoding: { type: "spki", format: "pem" },
  privateKeyEncoding: { type: "pkcs8", format: "pem" },
});
const secret = Buffer.from("AES key bytes...");
const wrapped = publicEncrypt({ key: publicKey, padding: constants.RSA_PKCS1_OAEP_PADDING, oaepHash: "sha256" }, secret);
const unwrapped = privateDecrypt({ key: privateKey, padding: constants.RSA_PKCS1_OAEP_PADDING, oaepHash: "sha256" }, wrapped);
          

// ECDSA (secp256k1)
import { generateKeyPairSync, createSign, createVerify } from "crypto";
const { publicKey: ecPub, privateKey: ecPriv } = generateKeyPairSync("ec", {
  namedCurve: "secp256k1",
  publicKeyEncoding: { type: "spki", format: "pem" },
  privateKeyEncoding: { type: "pkcs8", format: "pem" },
});
const msg = Buffer.from("I approve this action.");
const sign = createSign("SHA256"); sign.update(msg); sign.end();
const sig = sign.sign(ecPriv);
const verify = createVerify("SHA256"); verify.update(msg); verify.end();
console.log("valid:", verify.verify(ecPub, sig));

Slide 8 — Web3 Use Cases: Wallets, Smart Contracts, Signatures

Wallets manage asymmetric keys; applications request typed‑data signatures for explicit user intent; contracts verify or recover the signer. For confidentiality, encrypt data off‑chain with symmetric ciphers and wrap keys for recipients.

Slide 10 — Summary & Further Reading

Symmetric (AES‑GCM, ChaCha20‑Poly1305) delivers fast authenticated encryption; asymmetric (RSA‑OAEP, ECDSA) provides open verification and secure key exchange. Hybrid designs scale trust and performance. Enforce nonce uniqueness, domain‑separate signatures, rotate keys, validate parameters, and maintain audit trails.
